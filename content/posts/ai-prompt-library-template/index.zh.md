建 Prompt 库最大障碍不是工具，是不知道怎么组织。昨天让你整理 5 条，今天展示一个完整的真实模板。

## 目录结构

这个结构基于 Markdown 文件夹方案，你可以直接复制：

```text
prompt-library/
├── 写作类/
│   ├── 邮件.md
│   ├── 文章总结.md
│   └── ...
├── 分析类/
│   ├── 数据解读.md
│   ├── 案例拆解.md
│   └── ...
├── 日常类/
│   ├── 会议纪要.md
│   └── ...
└── README.md (全局说明)
```

每条 Prompt 的记录格式：

```markdown
## 原始提问
[你的 Prompt 文本]

## 效果评价
[1-5 分，或者简短评价]

## 迭代版本
### v2: [改进点]
- 新 Prompt
- 效果变化
```

## 示例 1：写作类——邮件 Prompt 的进化

**原始提问 v1**：

```
帮我写一封邮件给老板，说明项目进度。
```

**效果评价**：2/5 — 生成内容太笼统，没有具体格式，语气偏正式但不够得体。

**迭代版本 v2**：

```
你是一个项目经理，需要向技术总监汇报项目进度。写一封简洁的邮件，包含：1. 本周完成的功能 2. 遇到的技术难点 3. 下周计划。语气专业但不生硬，每部分不超过两句话。
```

**效果变化**：3/5 — 结构清晰，语气合适，但缺少邮件主题和行动呼吁。

**迭代版本 v3**：

```
你是一个项目经理，需要向技术总监汇报项目进度。写一封简洁的邮件，包含：1. 本周完成的功能 2. 遇到的技术难点 3. 下周计划。语气专业但不生硬，每部分不超过两句话。邮件主题用"项目进度汇报 - [项目名] - [日期]"，结尾加上"如需更多细节，我可以安排简短会议讨论。"
```

**最终评价**：5/5 — 完全满足需求，直接可用。

这条 Prompt 用了半年，从一句话到三段约束，每次修改都是基于实际使用中的痛点。

## 示例 2：分析类——数据解读

**原始提问**：

```
解释一下这份销售数据。
```

**效果评价**：3/5 — 给出了总体趋势，但缺少具体洞察和行动建议。

**迭代版本**：

```
你是一名数据分析师。我会提供一份 Excel 销售数据，包含日期、产品类别、销售额三列。请：
1. 找出销售额环比增长超过 20% 的月份
2. 识别销售额连续下滑的产品类别
3. 给出 3 条可执行的销售策略建议
用编号列表输出，每条建议不超过 30 字。
```

**RBGO 拆解**：
- **Role（角色）**：数据分析师
- **Background（背景）**：Excel 数据，包含日期、产品类别、销售额
- **Goal（目标）**：找出增长月份、下滑类别、给出策略
- **Output（输出）**：编号列表，每条建议不超过 30 字

**效果变化**：4/5 — 输出结构清晰，洞察具体，但有时会遗漏边缘情况。后续调整列名说明后，评分稳定在 4.5/5。

## 示例 3：日常类——会议纪要整理

**原始提问**：

```
把这段会议记录整理成纪要。
```

**效果评价**：3/5 — 场景明确但输出容易冗长、无重点，缺少结构约束。

**迭代版本**：

```
整理会议纪要，包含：1. 讨论要点（不超过 3 条） 2. 决议事项 3. 待办任务（负责人+截止时间）。决议和待办用加粗标注。
```

**效果变化**：5/5 — 加了字段约束后，输出格式稳定，直接发给参会人即可。

这条 Prompt 看起来简单，但每天用一次，每用一次都是效率提升。高频场景的 Prompt 值得花时间打磨。

## 小结

三个示例展示了 Prompt 库的核心价值：

1. 迭代记录让你看到进化路径，不必每次从零开始
2. RBGO 这样的框架帮你系统化思考，避免遗漏要素
3. 高频场景的简单 Prompt 也值得入库，因为复用价值高

Prompt 库建起来了，明天的毕业考核就是验收你四周积累的时候。
